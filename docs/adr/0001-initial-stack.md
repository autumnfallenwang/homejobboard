# 0001 — Initial tech stack

- **Status:** accepted
- **Date:** 2026-06-08
- **Deciders:** project founder (captured during `/devkit-init`)

## Context

This is the founding stack decision for the project, recorded at scaffold time. It captures what we picked and why, so future contributors can understand the constraints in play when these choices were made.

homejobboard aggregates jobs from the major job boards using pre-defined filters, scores newly-listed jobs for personal fitness with an LLM, and ranks them for quick review or applying. It is single-user and self-hosted only — there is no multi-tenancy and no auth — and it deploys to the existing home k3s cluster.

The dominant constraint shaping the stack is consistency with the sibling `homenews` / `homework` / `homecal` repos: the same TypeScript + pnpm + Turborepo monorepo, Hono + Zod API, Next.js App Router web, Drizzle/PostgreSQL, and Argo CD GitOps deployment. Reusing this proven shape lets the project share infrastructure (the `llmgw` LLM gateway, the k3s cluster, the GitOps pipeline) and tooling conventions with minimal new decisions.

## Decision

- **Primary language:** typescript
- **Package manager:** pnpm

- **Language:** TypeScript
- **Monorepo:** Turborepo + pnpm workspaces
- **Backend:** Hono + Zod (`apps/api`)
- **Frontend:** Next.js App Router (`apps/web`)
- **Shared types:** `packages/shared` (Zod schemas)
- **Database:** PostgreSQL via Drizzle ORM
- **LLM:** home cluster `llmgw` gateway (fitness scoring)
- **Lint/format:** Biome
- **Tests:** Vitest
- **Typecheck:** `tsc --noEmit`
- **Deploy:** Helm chart on home k3s cluster via Argo CD GitOps

## Consequences

**Positive:**

- Mirrors the sibling home* repos, so infrastructure (`llmgw`, k3s, GitOps) and conventions carry over with little new design.
- One language (TypeScript) end-to-end, with shared Zod schemas as the API↔web contract — no cross-language serialization friction.
- Biome + Vitest + tsc give a fast, unified check loop that the `devkit-*` skills drive.
- Single-user / no-auth scope keeps the surface small and lets the build focus on aggregation and scoring.

**Negative / trade-offs:**

- Inherits the family's operational complexity (k3s, Helm, Argo CD) even though this is a single-user app.
- TypeScript scrapers may be more awkward than Python for some boards' anti-bot or parsing needs.
- Coupling to the cluster's `llmgw` makes the fitness-scoring step dependent on that shared service.

**Open risks:**

- Job boards' terms of service and anti-scraping measures are unvalidated — board choice and API-vs-scrape are deferred to a POC.
- The LLM fitness-scoring approach (prompt, inputs, model) is undecided and may need iteration.
